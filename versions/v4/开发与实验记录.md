## v4 修复迭代 1

### 目标

修复 Context Gate 的监督链路、utility 标签、calibration 切分和 checkpoint 恢复约束；本轮只进行静态测试与 256 样本 Industrial smoke，不启动正式实验。

### 审计发现

1. `train_fuser.py` 使用独立 `utility_head` 预测 utility，utility BCE 的梯度不会进入生成 alpha 的 `ReliabilityFuser.gate`，因此“边际效用监督动态权重”没有真正实现。
2. `utility_label.py` 按 `text + 1.0 * id_residual` 生成标签，而 Industrial 的 v1 固定融合使用 `alpha0=0.5`，监督目标与比较基线不一致。
3. 旧示例将同一份 `valid.pt` 同时作为 fuser train/valid，存在模型选择泄漏，无法作为论文实验设置。
4. `scripts/common.py` 使用 `strict=False` 加载 v1 checkpoint，只打印 incompatible keys，结构不匹配时仍可能继续缓存并污染后续结果。
5. fuser checkpoint 未保存 `hidden_dim`、`dropout`、`dataset`、`seed`，加载时依赖命令行重新指定网络结构。
6. 现有 smoke 脚本覆盖部分基础性质，但缺少 utility-to-gate 梯度、按用户 calibration 隔离、dynamic=false 等价性和 checkpoint schema 的独立回归测试。
7. 当前 `hdrec` 环境未安装 `pytest`；Industrial 实际 v1 checkpoint 为 `versions/v1/outputs/Industrial_and_Scientific/deepseek-ai-DeepSeek-R1-Distill-Llama-8B_all_klw0.7_fa0.5/pytorch_model.bin`。

这些问题会使 gate 的方法描述与真实优化目标不一致、验证分数偏乐观，或在加载错误权重时仍产出缓存，均直接影响实验可复现性与论文可信度。

### 修改计划

1. 让同一个 gate logit 同时生成 utility probability 和 alpha；utility BCE 直接反传到 alpha gate，并记录梯度范数。
2. 按当前数据集 `alpha0` 生成 utility 标签并输出标签平衡统计。
3. 新增按用户、固定种子、尽量分层的 calibration 切分脚本，保存样本索引和统计。
4. 对 v1 checkpoint 启用 incompatible-key 硬失败；完善 fuser checkpoint 自描述字段和无命令行结构恢复。
5. 增加独立 pytest 测试并运行 CPU 测试；通过后再以 `screen` 启动 Industrial 256 样本 cache、Oracle 和 utility analysis。

说明：总损失采用 `rank_loss + lambda_u * utility_loss + lambda_r * shrink_loss`。原修复提示中的减号若用于最小化会鼓励增大 BCE 和 shrink loss，与设计目标相反。

### 核心实现

```text
gate_logit = gate(normalized_features)
utility_prob = sigmoid(gate_logit)
alpha = clamp(alpha0 + rho * (2 * utility_prob - 1), 0, alpha_max)
final_logits = text_logits + alpha * id_residual
```

训练损失为：

```text
loss = rank_loss + utility_loss_weight * BCEWithLogits(gate_logit, utility_label)
                 + shrink_loss_weight * mean((alpha - alpha0)^2)
```

每轮首个 batch 分别记录 `grad(alpha_gate)` 与 `grad(utility_loss -> gate)`。后者由 `autograd.grad(utility_loss, gate.parameters())` 独立测量，用于证明 utility 监督确实进入 alpha gate。

3. fuser checkpoint 未保存 feature schema、基础 checkpoint、fusion temperature 和 utility target，评估无法证明 fuser 与 cache 来自同一冻结模型和同一监督定义。

5. Oracle 步长仍默认为 0.1，未输出全局 alpha 曲线、calibration-valid 最佳 global alpha、逐样本 best alpha 和 Utility×alpha 交叉表。
6. fuser 训练日志未拆分 rank/utility/shrink loss，也缺少 Rank Utility AUC 与 alpha 边界占比。

### Alpha Target 构造

- 统一网格：`0.00, 0.05, ..., 1.00`，共 21 档。
- 仓库 `target_ranks` 返回 0-based rank；因此 reciprocal rank 为 `1/(rank+1)`，DCG 为命中 K 时 `1/log2(rank+2)`。

- CE soft target：`softmax(-CE / tau_ce)`，首轮 `tau_ce=0.5`。
- Metric soft target：先对每个样本内部 CE 标准化，再使用 `DCG@10 + 0.1*RR - 0.05*normalized_CE`，以 `tau_metric=0.1` softmax。

### 新假设

同一序列内，ID residual 可能同时增强有用候选和错误候选。序列级 alpha 无法区分两者；候选级非负 gate 可能保留可靠修正并抑制误导修正，同时严格保持 `final = text + max_alpha * gate * raw_id_residual`。

### 候选级机制诊断

- calibration-train/valid 分别包含 8495/2128 个用户样本、4971 个候选物品，用户无交叉。
- 缓存包含 text/ID logits、labels、user_ids、history、10 维 sequence features 和 fusion temperature，可直接离线诊断。
- 第一版候选特征维度为 30。若将 train 全量预展开为 float32，需要约 `8495*4971*30*4 = 5.06 GB`（约 4.71 GiB），还不含 logits、梯度和中间张量，因此实现必须按 sample batch 与 candidate chunk 在线生成。

### 简单规则结果

- calibration-train 在预注册 P2-P6 与 max_alpha 网格上选出 `P6 Soft Consensus, max_alpha=0.1`，train NDCG@10=0.146784。

- Policy-Family Oracle NDCG@10=0.166090，相对 text-only 有 +0.009104 空间；更严格的 Constrained Candidate Policy Oracle=0.164297。二者均只是在预定义策略中按样本使用标签选择的诊断上界，不是可实现方法结果。

### Candidate Gate 设计

- Go/No-Go-1 因 Policy-Family Oracle 空间成立，允许进入 Linear Candidate Gate；固定规则本身不成立。
- 候选特征共 30 维：18 维候选分数/概率/rank/一致性，广播但不复制存储的 10 维 sequence context，以及 2 维只由训练交互计算的 item popularity。特征接口不接收 target label。
- Linear Gate 仅为 `sigmoid(Linear(30,1))`，参数量 31；初始 probability=0.01。只有 Linear 在 gate_dev 高于 text-only 才允许训练 `30->16->1` Tiny MLP。
- Gate 只能生成 `[0,1]` 系数并乘原始非负 ID residual，不能修改文本 logits、添加独立分数或学习 user/item embedding。
- 全量 train 候选特征预展开约需 5.06 GB（4.71 GiB）；实际使用 sample batch=16/64、candidate chunk=1024，训练阶段单块候选特征约 1.97 MB，不保存三维候选特征文件。

- v4 相对 v1 fixed 的 NDCG@10 为 `+0.003554`（`+2.58%`），Recall@10 为 `+0.005442`（`+3.23%`），MRR 为 `+2.37%`。相对 text-only，NDCG@10 为 `+0.000787`（`+0.56%`），但 Recall@10 为 `-0.000451`（`-0.26%`）。
- v4 相对 text-only 的 NDCG@10 配对 bootstrap 均值差为 `+0.000787`，95% CI=`[0.000313, 0.001266]`，`P(diff>0)=1.0`；相对 v1 fixed 的均值差为 `+0.003554`，95% CI=`[0.002887, 0.004271]`，`P(diff>0)=1.0`。在当前 test 上，两项比较均具有 strong evidence。
- Oracle-Select 的 NDCG@10=`0.148613`，Oracle-Alpha 的 NDCG@10=`0.151110`；相对 text-only 的提升分别为 `+5.69%` 和 `+7.47%`。v4 当前恢复了约 `7.5%` 的 Oracle-Alpha NDCG@10 提升空间，说明仍有较大可预测空间。
- Oracle-Alpha 的最优 alpha 仍明显两端化：alpha=0 占约 `49.91%`，alpha=1 占约 `27.98%`，两端合计约 `77.89%`。这支持后续研究“关闭或强开启 ID 修正”的离散/选择性策略，但不能直接把 Oracle 当作可实现结果。
- 最终 v4 Gate 的 harm rate=`0.491890`、benefit rate=`0.352063`，gate mean=`0.466115`，gate>0.5 比例=`0.437903`。相比 Industrial，Arts 的 Gate 已表现出更明显的样本级选择性，但 harm rate 较高。

### Arts 结果判断

- Arts 上 v4 首次同时超过 text-only 和 v1 fixed 的 NDCG@10、NDCG@5、Recall@5 与 MRR；相对 v1 fixed 的提升具有稳定统计证据。
- Recall@10 尚未超过 text-only，说明当前 Gate 更主要改善排序质量和前部命中，而不是所有 K 值都同步提升。
- calibration-valid 的严格 Candidate Gate 判定仍为 `No-Go`，主要原因是 harm rate 未满足预设的安全阈值；最终 test 的正向结果应作为有希望的跨数据集证据，而不是已经完成方法验证。
- Industrial 与 Arts 的共同结论是：固定 alpha 会引入伤害，动态融合能够回收一部分收益；但 Oracle 与实际 Gate 之间仍存在明显差距。下一步应优先围绕两端 alpha 选择、harm-aware 目标和跨数据集稳定性开展消融，再决定是否进入更复杂的 v5 表示对齐方向。
